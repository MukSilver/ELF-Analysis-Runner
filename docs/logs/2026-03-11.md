## 오늘 배운 것
- sys.argv는 터미널에서 전달한 인자를 가져오는 데 사용된다.
- main()은 프로그램의 전체 흐름을 관리하는 중심 함수 역할을 한다.
- Path(path)를 사용하면 파일 존재 여부, 파일 여부, 열기 등을 더 편하게 처리할 수 있다.
- ELF 파일은 앞 4바이트의 매직 넘버 b"\x7fELF" 로 판별할 수 있다.
- 파이썬에서는 예외가 발생할 수 있는 작업을 try / except로 감싸 안전하게 처리할 수 있다.
- Windows에서 만든 실행 파일은 보통 PE 형식이고, WSL/Linux에서 만든 실행 파일은 ELF 형식이다.
- C의 argc는 전달된 인자의 개수이고, argv는 전달된 인자 문자열 목록이다.
- argv[2][0]은 세 번째 인자 문자열의 첫 번째 문자를 꺼내는 표현이다.
- gcc ... -o ...의 -o 옵션은 출력 파일 이름을 지정하는 역할을 한다.
- 리눅스 명령어 /bin/ls, /bin/cat는 실제로 ELF 실행 파일이다.

## 막힌 점
- try / except가 C의 어떤 개념과 대응되는지 처음에 감이 잘 안 잡혔음
- argv, argc, argv[2][0] 같은 인자 처리 방식이 처음엔 헷갈렸음
- /bin/ls로 테스트했는데 파일이 없다고 떠서 코드 문제인지 환경 문제인지 혼동됨
- WSL에서 파이썬이 안 깔린 줄 알고 실행 환경이 헷갈렸음
- Windows 터미널과 WSL에서 python / python3 사용 차이 때문에 실행이 꼬였음
- C 샘플을 윈도우에서 빌드하면서 .exe, .obj, .pdb, .ilk 같은 파일이 한꺼번에 생겨 폴더가 지저분해짐
- 윈도우에서 만든 실행 파일이 ELF인지, 리눅스에서 만든 실행 파일이 ELF인지 구분이 처음엔 명확하지 않았음
- -o 옵션 의미와 ./samples/bin/calc에서 ./ 의미가 헷갈렸음

## 해결한 점
- try / except는 반복문이 아니라, C에서 if로 에러를 직접 확인하던 흐름을 파이썬에서 예외 처리로 감싼 것이라는 감각을 잡음
- argc는 전달된 인자 개수, argv는 인자 문자열 목록이라는 점을 이해했고, argv[2][0]은 세 번째 문자열의 첫 글자를 꺼내는 표현이라는 걸 이해함
- /bin/ls가 안 잡힌 건 코드 문제가 아니라 윈도우 환경에서 리눅스 경로를 쓴 것 때문이라는 걸 파악함
- WSL에서는 python3, 윈도우 PowerShell에서는 python을 써야 한다는 환경 차이를 직접 확인함
- WSL에서 /bin/ls를 분석해 실제로 ELF 판별이 정상 동작하는 것을 확인함
- 리눅스 시스템 ELF 샘플과 직접 만든 C 기반 ELF 샘플, 비정상 종료용 샘플까지 준비해 테스트 기반을 갖춤
- C 소스와 바이너리 결과물을 분리해야 한다는 필요성을 느끼고, samples/c, samples/bin 식으로 정리하는 방향을 잡음
- 윈도우에서 빌드한 hello.exe는 PE이고, WSL/Linux에서 gcc로 만든 바이너리가 ELF라는 점을 이해함
- -o는 출력 파일 이름 지정 옵션이고, ./는 현재 폴더 기준 경로라는 점을 이해함
